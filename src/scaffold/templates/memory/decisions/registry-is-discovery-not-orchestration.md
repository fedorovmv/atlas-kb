# Решение: Registry is discovery, not orchestration

## Контекст

Продукту нужен способ для агентов и инструментов обнаруживать доступные метаданные внутри Synapse API Mesh.

## Проблема

Если ответственность реестра не определена явно, его можно перепутать с runtime-оркестратором, который выбирает агенты или выполняет вызовы.

## Решение

Реестр — это компонент обнаружения. Он хранит и предоставляет метаданные и доступность с фильтрацией по правам доступа. Он не выбирает целевые агенты и не выполняет runtime-вызовы.

## Обоснование

Эта граница сохраняет простоту компонента и избегает смешивания обнаружения метаданных, бизнес-решений и runtime-исполнения.

## Рассмотренные альтернативы

- Реестр только как хранилище метаданных (без фильтрации)
- Реестр как runtime-оркестратор

## Отклонённые альтернативы

### Реестр как runtime-оркестратор

Статус: отклонено.

Причина: это смешивает ответственность обнаружения и runtime, создавая неверные ожидания относительно реестра.

**Технические ограничения:** Реестр не имеет контекста вызова (права, состояние сессии, приоритеты), необходимого для принятия runtime-решений. Добавление оркестрации потребовало бы состояния сессии и логики диспетчеризации, которых в компоненте нет по дизайну.

**Компромиссы относительно выбранного варианта:** Выбранный вариант (только обнаружение) сохраняет компонент stateless и тестируемым в изоляции. Вариант с оркестрацией уменьшает количество компонентов в системе, но переносит бизнес-логику выбора в инфраструктурный слой, что затрудняет тестирование и изменение правил выбора.

**Почему не подходит контексту проекта:** API Mesh предполагает множественные источники прав и политик доступа; централизация выбора в реестре связала бы инфраструктуру с конкретными правилами доступа, блокируя независимую эволюцию этих слоёв.

## Последствия

- Выбор агента остаётся за пределами реестра.
- Runtime-проверки должны описываться отдельно.
- Изменения кода вокруг реестра должны сохранять эту границу, пока новое решение её не заменит.

## Свидетельства текущего поведения

- FilterCardsForCaller в internal/registry/access_filter.go обеспечивает поведение только для обнаружения.

## Примеры использования

- `FilterCardsForCaller` в `internal/registry/access_filter.go` вызывает реестр для получения отфильтрованного списка метаданных; последующий выбор агента выполняется вызывающей стороной, не реестром.
- Клиентское приложение запрашивает доступные инструменты через реестр, получает список с фильтрацией по правам, затем вызывает целевой агент напрямую.
